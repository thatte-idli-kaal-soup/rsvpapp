* The MongoDB 8.x + kernel 6.19 gotcha                              :reference:

This host's kernel (=uname -r= → =7.0.0-22-generic=) is read by MongoDB as ≥ 6.19.
Per MongoDB =SERVER-121912=: MongoDB 8.0+ crashes on startup on kernel 6.19+ due to
the vendored TCMalloc violating the rseq ABI. Affects ALL 8.x packages (website,
package managers, Docker) — bumping the 8.x point version does NOT help.

Symptoms seen, both the same root cause:
- =mongo:latest= (8.2) → SIGSEGV, exit code 139, dies seconds after "Waiting for connections".
- =mongo:8.0= → clean fatal refusal: "Linux kernel versions 6.19 and newer has a known incompatibility".

Workarounds:
- *Use =mongo:7.0=* (predates the broken allocator) — what we did.
- Or verify with =bsondump= (no server needed at all).
- Or downgrade the host kernel to ≤ 6.18 (overkill here).

The =mongodump=/=mongorestore= *tools* are NOT affected — only the *server* is. So
backups themselves are fine regardless of kernel.

* Future-useful commands                                                :howto:

** Restore from the gpg archive

Decrypt, then restore. Decrypt is a separate step (prompts for the passphrase):

#+begin_src sh
gpg -d rsvp-2026-06-11.archive.gpg > rsvp-2026-06-11.archive
#+end_src

Then restore into a running local Mongo (see next section for spinning one up).
The =--nsFrom/--nsTo= renames the cryptic db to something sane:

#+begin_src sh
<TOOLS>/mongorestore \
  --uri="mongodb://localhost:27017" \
  --archive=rsvp-2026-06-11.archive --gzip \
  --nsFrom="heroku_cdbwn214.*" --nsTo="rsvpdata.*"
#+end_src

Tip: delete the decrypted plaintext archive when done; keep only the =.gpg=.

** Spin up a working local MongoDB (survives restarts)              :docker:

=mongo:7.0= to dodge the kernel bug; a *named volume* so data persists across
container stop/rm/recreate.

#+begin_src sh
docker volume create rsvp-store
docker run -d --name rsvp-db -p 27017:27017 -v rsvp-store:/data/db mongo:7.0
docker logs -f rsvp-db   # wait for "Waiting for connections", then C-c
#+end_src

Everyday use afterwards:

#+begin_src sh
docker start rsvp-db    # bring it up
docker stop rsvp-db     # put it away — data stays in the rsvp-store volume
#+end_src

Recreate the container any time (data is in the volume, not the container):

#+begin_src sh
docker rm -f rsvp-db
docker run -d --name rsvp-db -p 27017:27017 -v rsvp-store:/data/db mongo:7.0
#+end_src

NOTE: the volume is tied to THIS machine + Docker install. The =.gpg= archive is the
portable copy — to move machines, copy the archive and re-restore there.

* Calendar delete (already run — kept for reference) ☠           :destructive:
:PROPERTIES:
:header-args:sh: :eval no
:END:

Run from inside =fly ssh console -a tiks-ultimate-rsvp=. Lookup-only (does not
recreate). Needs the service account to be the calendar OWNER, else 403.

#+begin_src python :eval no
from rsvp.gdrive_utils import create_service
CALENDAR_TITLE = "RSVP App Events Calendar"   # confirm exact summary
svc = create_service("calendar")
page_token, cal_id = None, None
while True:
    cl = svc.calendarList().list(pageToken=page_token).execute()
    for e in cl["items"]:
        if e["summary"] == CALENDAR_TITLE:
            cal_id = e["id"]; break
    page_token = cl.get("nextPageToken")
    if cal_id or not page_token:
        break
if not cal_id:
    print("Not found; nothing to delete.")
else:
    svc.calendars().delete(calendarId=cal_id).execute()
    print("Deleted", cal_id)
#+end_src
